### Findings

#### [FIXED] Grafana datasource UID mismatch — high

Symptom: every Overview / Dispatch panel rendered empty. `API RPS`, `Error Rate`, `Latency (p50/p95/p99)`, `Active drivers` all blank.

Root cause: provisioned dashboards reference `datasource.uid: "prometheus"` (and `"loki"`), but `infra/observability/grafana/provisioning/datasources/datasources.yml` did not pin UIDs. Grafana auto-generated random UIDs (e.g. `PBFA97CFB590B2093`). Dashboard JSON couldn't resolve queries.

Fix (this session): added `uid: prometheus` and `uid: loki` to provisioning, restarted Grafana container, panels populate.

File: `infra/observability/grafana/provisioning/datasources/datasources.yml`.

#### [FIXED] OSRM not host-exposed in dev compose

Doc said `localhost:5000`; container only exposes internally. Probe via `docker exec rcab-dev-api-1 node -e "fetch('http://osrm:5000/...')"`. Documented in updated walkthrough (pending).

#### [FIXED] health-ready int spec async listen race

`mockOsrm.listen(0)` returned synchronously and `address()` was read before the `listening` event fired → port `0` → probe to `http://127.0.0.1:0` failed → readiness returned 503. Fix: await `listening` promise before reading port.

File: `apps/api/test/integration/health-ready.int.spec.ts`.

#### [PARTIALLY MITIGATED] BullMQ shutdown noise breaks suite-level pass — med

Symptom: `drivers.int.spec.ts` + `realtime-location.int.spec.ts` show as `FAIL` at the file level even though every individual test passes (88/88 across the suite).

Root cause: when `app.close()` runs, BullMQ Worker / shared Queue ioredis client closes its socket. Pending blocking-fetch and queue commands reject with `Connection is closed.` after the JS-level shutdown is already done. Vitest captures the rejection via a Node-native dispatch path that does NOT go through `process.on('unhandledRejection')` or `process.emit('unhandledRejection', ...)` — confirmed by intercepting both and observing the rejection still reaches Vitest's reporter.

Mitigation applied this session:
- Set `autorun: false` on `DispatchProcessor` and `PoolExpireProcessor` decorators gated on `process.env.RCAB_DISABLE_BULL_AUTORUN === '1'`.
- `test/setup.int.ts` sets the env var.
- Added `RealtimeGateway.onModuleDestroy` to quit the duplicated adapter pub/sub clients explicitly.

Residual: the BullModule.forRoot shared connection still emits one rejection per spec on teardown. Vitest reports 2 files failed, exit code 1.

Open question: drop the AppModule import in those 2 specs and build a minimal Nest module per spec, OR file a BullMQ upstream issue and gate CI on "tests passed" rather than "exit code", OR live with 2-file noise until BullMQ 6.x.

Files: `apps/api/test/setup.unhandled.ts`, `apps/api/test/setup.int.ts`, `apps/api/vitest.config.int.ts`, `apps/api/src/modules/dispatch/dispatch.processor.ts`, `apps/api/src/modules/matching/pool-expire.processor.ts`, `apps/api/src/modules/realtime/realtime.gateway.ts`.
