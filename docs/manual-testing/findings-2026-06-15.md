Symptom: every Overview / Dispatch panel rendered empty. `API RPS`, `Error Rate`, `Latency (p50/p95/p99)`, `Active drivers` all blank.

#### [FIXED] health-ready int spec async listen race

File: `apps/api/test/integration/health-ready.int.spec.ts`.

#### [PARTIALLY MITIGATED] BullMQ shutdown noise breaks suite-level pass ‚Äî med

Symptom: `drivers.int.spec.ts` + `realtime-location.int.spec.ts` show as `FAIL` at the file level even though every individual test passes (88/88 across the suite).

Root cause: when `app.close()` runs, BullMQ Worker / shared Queue ioredis client closes its socket. Pending blocking-fetch and queue commands reject with `Connection is closed.` after the JS-level shutdown is already done. Vitest captures the rejection via a Node-native dispatch path that does NOT go through `process.on('unhandledRejection')` or `process.emit('unhandledRejection', ...)` ‚Äî confirmed by intercepting both and observing the rejection still reaches Vitest's reporter.

Mitigation applied this session:
- Set `autorun: false` on `DispatchProcessor` and `PoolExpireProcessor` decorators gated on `process.env.RCAB_DISABLE_BULL_AUTORUN === '1'`.
- `test/setup.int.ts` sets the env var.
- Added `RealtimeGateway.onModuleDestroy` to quit the duplicated adapter pub/sub clients explicitly.

Residual: the BullModule.forRoot shared connection still emits one rejection per spec on teardown. Vitest reports 2 files failed, exit code 1.

Open question: drop the AppModule import in those 2 specs and build a minimal Nest module per spec, OR file a BullMQ upstream issue and gate CI on "tests passed" rather than "exit code", OR live with 2-file noise until BullMQ 6.x.

Files: `apps/api/test/setup.unhandled.ts`, `apps/api/test/setup.int.ts`, `apps/api/vitest.config.int.ts`, `apps/api/src/modules/dispatch/dispatch.processor.ts`, `apps/api/src/modules/matching/pool-expire.processor.ts`, `apps/api/src/modules/realtime/realtime.gateway.ts`.

## Demo 1 ‚Äî I am a real user

Walked with Firebase test phone `+919999999999` / OTP `123456` to skip SMS.

### Findings

#### [FIXED] No CORS on the API ‚Äî high

Symptom: web sign-in form ‚Üí `signInWithPhoneNumber` succeeded on Firebase, but the immediate `POST /v1/auth/firebase-exchange` failed with `Failed to fetch` in the browser. No request was visible in the api access log either (preflight rejected).

Root cause: `apps/api/src/main.ts` never called `app.enableCors()`. NestJS default is same-origin only; web at `:3002` ‚Üí api at `:3000` is cross-origin.

Fix: `enableCors({ origin: CORS_ORIGINS.split(','), credentials: true })`. Default origins cover `:3001` and `:3002`. `credentials: true` so the refresh-token cookie roundtrip works.

File: `apps/api/src/main.ts`.

#### [FIXED] No `.env.local` template for web Firebase config

Symptom: `apps/web/.env.local` did not exist; no `.env.example` either. Demo 1 instructions silently assumed the operator already knew which `NEXT_PUBLIC_FIREBASE_*` vars existed.

Fix this session: pasted Firebase web SDK config into `apps/web/.env.local` and restarted the web container. The file is gitignored.

Follow-up to file: add `apps/web/.env.example` with the 4 required `NEXT_PUBLIC_FIREBASE_*` keys, document the Console steps in the walkthrough.

#### [FIXED] AuthController + AuthService DI dropped under tsx watch ‚Äî high

Symptom: every `/v1/auth/firebase-exchange` returned **500** with body `{code: internal_error}`. The AppExceptionFilter swallowed the underlying error silently, so initial triage looked like a Firebase verification issue. Adding a 5xx log line surfaced:

```
TypeError: Cannot read properties of undefined (reading 'exchangeFirebaseToken')
    at AuthController.firebaseExchange (auth.controller.ts:38:36)
```

Then, after fixing controller-level DI, the same shape at the service:

```
TypeError: Cannot read properties of undefined (reading 'verifyIdToken')
```

Root cause: `apps/api` dev mode runs via `tsx watch`, which uses esbuild for the TypeScript transform. esbuild does NOT emit `design:paramtypes` decorator metadata. NestJS DI then can't resolve the constructor types and silently assigns `undefined` to every implicit constructor parameter. Other controllers happened to work because their guards short-circuited before the missing instance was touched.

Fix: explicit `@Inject(<Token>)` on every constructor param of `AuthController` and `AuthService`. Permanent fix would be to switch tsx to a transform that emits metadata (swc with `transform.decoratorMetadata: true`, or fall back to `tsc --watch`), tracked separately.

Files: `apps/api/src/modules/auth/auth.controller.ts`, `apps/api/src/modules/auth/auth.service.ts`.

#### [FIXED] AppExceptionFilter ate 5xx exceptions silently

The catch-all returned `{code: internal_error}` to the client but logged nothing server-side. Cost time during the DI hunt above. Fixed to log at error level via NestJS Logger so the pino transport records it.

File: `apps/api/src/common/filters/app-exception.filter.ts`.

#### [FIXED] React StrictMode broke invisible reCAPTCHA on `/signin`

Symptom: after solving the (invisible) reCAPTCHA challenge, the page threw `Cannot read properties of null (reading 'style')` from inside `grecaptcha`, blocking the OTP submission.

Root cause: the `RecaptchaVerifier` was constructed in a `useEffect` on mount with a cleanup that called `verifier.clear()`. React StrictMode runs every effect twice in dev (mount ‚Üí cleanup ‚Üí mount); the first cleanup nulled out the DOM container the recaptcha JS still referenced.

Fix: lazy-init the verifier inside the form submit handler. Runs at most once per page lifetime after the container ref is stable, and never gets torn down mid-flow.

File: `apps/web/src/app/signin/sign-in-client.tsx`.

#### [FIXED] `react-leaflet` + `leaflet` not installed in the web container

Symptom: after sign-in success, the router pushed to `/book`, which failed to compile with `Module not found: Can't resolve 'react-leaflet'`. Package listed in `apps/web/package.json` but missing from the running container's `node_modules`.

Root cause: `apps/web/node_modules` is an anonymous volume in `docker-compose.dev.yml`. The volume was populated before `react-leaflet` was added to the deps; subsequent `pnpm install` on the host never touched it.

Fix: `docker exec rcab-dev-web-1 pnpm install --filter=@rcab/web`, then `docker restart rcab-dev-web-1`. Anonymous volume is now in sync.

Follow-up: `pnpm dev:up` should detect lockfile drift vs. the volume contents on every up ‚Äî log a follow-up RCAB-E1.S2 ticket.
